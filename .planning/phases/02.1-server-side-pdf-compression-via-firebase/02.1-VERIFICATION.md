## Deviations from Original Plan

### SCOMP-02 Deviation — Preset Implementation (Documented, Creates Label Inaccuracy)

The original plan and requirement SCOMP-02 specified using Ghostscript `-dPDFSETTINGS` named presets (`/prepress` at 300 DPI, `/ebook` at 150 DPI, `/screen` at 72 DPI). During execution, the user discovered that `-dPDFSETTINGS` overrides custom DPI flags, so the implementation was changed to fully explicit DPI + QFactor parameters.

**What changed:**
- Backend: `GS_PRESETS.best` = 150 DPI / QFactor 0.30 (not 300 DPI)
- Backend: `GS_PRESETS.balanced` = 144 DPI / QFactor 0.18 (not 150 DPI)
- Backend: `GS_PRESETS.compressed` = 36 DPI / QFactor 0.90 (not 72 DPI)
- Frontend: DPI labels still say "300 DPI", "150 DPI", "72 DPI" — not updated to match new values

The deviation is sound from an engineering perspective (hitting target compression ratios was verified by the user) but the frontend DPI labels are now misleading. REQUIREMENTS.md SCOMP-02 still references the original values.

### Additional Features Beyond Plan

- **Rate limiting**: In-memory sliding window (5 req/min per IP, 30 req/min global) — user-requested, not in original plan
- **Size guard**: Returns original PDF if compression inflates file — user-requested, not in original plan
- **`maxInstances: 2`**: Caps concurrent Cloud Function instances — user-requested

These are all beneficial additions and do not create gaps.

## Gaps Summary

Two verified truths are PARTIAL rather than FAILED — the functionality works end-to-end, but there is a label accuracy issue:

**Root cause:** During iterative preset tuning in Plan 01, the user requested switching from Ghostscript named presets (`/prepress`, `/ebook`, `/screen`) to fully explicit DPI + QFactor parameters to hit specific compression ratio targets. This was a valid engineering decision. However, the frontend DPI labels ("300 DPI", "150 DPI", "72 DPI") were not updated to match the new actual values (150 DPI, 144 DPI, 36 DPI). REQUIREMENTS.md SCOMP-02 also still references the original approach.

**Impact:** Users see misleading DPI values. A user uploading to "Best Quality (300 DPI)" receives output at 150 DPI. This is not a functional blocker — the compression works and text is preserved — but it is a documentation and UI accuracy issue.

**Recommended fix options (pick one):**
1. Update `index.html` preset descriptions to show actual DPI values: "Best Quality (150 DPI)", "Balanced (144 DPI)", "Compressed (36 DPI)"
2. Update REQUIREMENTS.md SCOMP-02 to say "three presets with tuned DPI/QFactor" instead of specifying /prepress, /ebook, /screen
3. Accept the UI labels as approximate/marketing labels (e.g., "up to 300 DPI quality") and add a note in requirements

The phase goal — "replace client-side Heavy mode with server-side Ghostscript compression, preserving text selectability while achieving effective image downsampling across three quality presets" — is **substantively achieved**. The label inaccuracy is the only gap.
