# Phase 02.1: Server-Side PDF Compression via Firebase - Research

**Researched:** 2026-03-03
**Domain:** Firebase Cloud Functions (2nd gen), Ghostscript, multipart binary HTTP, CORS
**Confidence:** HIGH for Ghostscript integration and Cloud Functions architecture. MEDIUM for specific request size limits (conflicting sources). LOW for emulator behavior with binary uploads (limited verified documentation).

---

## Summary

Phase 02.1 replaces the existing client-side Heavy mode (which rasterizes pages to JPEG, destroying text selectability) with a server-side Ghostscript compression pipeline hosted on Firebase Cloud Functions. Ghostscript's `-dPDFSETTINGS` presets downsample only raster images while preserving text vectors and font data, making text remain selectable in the output PDF — the key quality advantage over the current approach.

Ghostscript 9.55.0 is confirmed available as a system package in the Cloud Functions Ubuntu 22.04 runtime. No binary bundling is needed. The function receives a PDF via multipart/form-data (using Busboy — not Multer, which is broken in Cloud Functions), writes it to `/tmp`, spawns `gs` as a child process with the appropriate preset flags, reads the output file, and returns it as `application/pdf` binary. The frontend calls this endpoint with `fetch` using `FormData`, receives an `ArrayBuffer`, and handles the result exactly like the existing Fast mode output.

The architecture adds a new `functions/` directory alongside `index.html` and requires a Firebase project (Blaze plan required to deploy Cloud Functions; Spark/free plan blocks deployment). The existing Fast mode and all other tools remain untouched. Only the `compressFileHeavy()` function path in `index.html` is replaced with a `fetch` call to the Cloud Function endpoint.

**Primary recommendation:** Use Firebase Cloud Functions 2nd gen with a single HTTP `onRequest` function that receives PDF via Busboy-parsed multipart upload, runs `gs` from the system path, returns compressed PDF binary. Configure `memory: '512MiB'` and `timeoutSeconds: 300` for safety. Use `cors: true` in the `onRequest` options — the simplest correct approach for a single-origin open tool.

## Standard Stack

### Core

| Library | Version | Purpose | Why Standard |
|---------|---------|---------|--------------|
| firebase-functions | v6+ (2nd gen) | HTTP trigger that receives PDF upload, runs Ghostscript, returns compressed PDF | Official Firebase SDK; 2nd gen is the currently recommended version for all new functions |
| firebase-tools | v13+ | CLI for `firebase init`, `firebase deploy`, `firebase emulators:start` | Official Firebase deployment toolchain |
| busboy | v1.6+ | Multipart/form-data parser inside Cloud Functions | Multer is broken in Cloud Functions (body already parsed by middleware); busboy is the documented workaround |
| Node.js child_process | built-in | Spawn `gs` binary as a subprocess | Ghostscript is a native binary; child_process.execFile is the standard Node.js way to invoke it |

### Supporting

| Library | Version | Purpose | When to Use |
|---------|---------|---------|-------------|
| cors (npm) | v2.x | CORS headers for browser fetch requests | Needed if using express wrapper; with onRequest cors option, may not be needed separately |
| os (built-in Node) | built-in | `os.tmpdir()` for temp file path | Required for writing uploaded PDF to disk before Ghostscript processes it |
| fs/promises (built-in Node) | built-in | Read/write temp files, clean up after processing | Standard file I/O for the temp file lifecycle |
| uuid or crypto.randomUUID | built-in (Node 19+) | Generate unique temp file names | Prevent collisions when concurrent invocations process different PDFs simultaneously |

### Alternatives Considered

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| Ghostscript system binary | compress-pdf npm package | compress-pdf downloads its own Ghostscript binary during npm install — cleaner for local dev, but adds 50+ MB to deployment and is redundant since Ghostscript is already in the runtime. Use system `gs` directly. |
| Multipart upload direct to Cloud Function | Client uploads to Cloud Storage, function reads from bucket | Avoids request size limits entirely, but adds Storage dependency, requires bucket setup, and complicates the client-side flow. Not worth it for PDFs under the size limit. |
| Cloud Functions 2nd gen (HTTP) | Cloud Run custom container | Cloud Run gives more control (custom Dockerfile, background processes) but requires container registry setup, more operational complexity, and is overkill for a single function. Cloud Functions 2nd gen built on Cloud Run is simpler. |
| Binary multipart POST | Base64-encoded JSON body | Base64 inflates payload 33%. For a 10 MB PDF that becomes 13.3 MB, approaching limits. Multipart is the correct approach for binary uploads. |

### Pattern 2: HTTP Cloud Function — Receive PDF, Compress with Ghostscript, Return PDF

**What:** Single `onRequest` function accepting multipart POST, running `gs`, returning compressed PDF binary.
**When to use:** Core of the phase — this is the entire server-side function.

```javascript
// Source: firebase-functions v2 API + busboy + Node.js child_process
// functions/index.js

const { onRequest } = require('firebase-functions/v2/https');
const busboy = require('busboy');
const { execFile } = require('child_process');
const { promisify } = require('util');
const os = require('os');
const path = require('path');
const fs = require('fs/promises');
const crypto = require('crypto');

const execFileAsync = promisify(execFile);

// Ghostscript preset mapping for the three user-facing presets
const GS_PRESETS = {
  best:       '/prepress',   // 300 DPI — Best Quality
  balanced:   '/ebook',      // 150 DPI — Balanced
  compressed: '/screen'      // 72 DPI  — Compressed
};

exports.compressPdf = onRequest(
  {
    cors: true,
    memory: '512MiB',
    timeoutSeconds: 300,    // 5 minutes — large PDFs may take time
    region: 'us-central1'  // default region
  },
  async (req, res) => {
    if (req.method !== 'POST') {
      return res.status(405).send('Method Not Allowed');
    }

    // Parse multipart form data using busboy
    let pdfBuffer = null;
    let preset = 'balanced'; // default

    await new Promise((resolve, reject) => {
      const bb = busboy({
        headers: req.headers,
        limits: { fileSize: 50 * 1024 * 1024 } // 50 MB limit
      });

      bb.on('file', (fieldname, file, info) => {
        const chunks = [];
        file.on('data', chunk => chunks.push(chunk));
        file.on('end', () => {
          pdfBuffer = Buffer.concat(chunks);
        });
      });

      bb.on('field', (fieldname, value) => {
        if (fieldname === 'preset') preset = value;
      });

      bb.on('close', resolve);
      bb.on('error', reject);

      // Cloud Functions provides req.rawBody for multipart
      if (req.rawBody) {
        bb.end(req.rawBody);
      } else {
        req.pipe(bb);
      }
    });

    if (!pdfBuffer || pdfBuffer.length === 0) {
      return res.status(400).json({ error: 'No PDF file received' });
    }

    const gsPreset = GS_PRESETS[preset] || GS_PRESETS.balanced;

    // Write input to /tmp with a unique name to handle concurrent requests
    const id = crypto.randomUUID();
    const inputPath = path.join(os.tmpdir(), `${id}-input.pdf`);
    const outputPath = path.join(os.tmpdir(), `${id}-output.pdf`);

    try {
      await fs.writeFile(inputPath, pdfBuffer);

      // Invoke Ghostscript — system binary is available at /usr/bin/gs
      await execFileAsync('gs', [
        '-q',
        '-dNOPAUSE',
        '-dBATCH',
        '-dSAFER',
        '-sDEVICE=pdfwrite',
        '-dCompatibilityLevel=1.4',
        `-dPDFSETTINGS=${gsPreset}`,
        '-dEmbedAllFonts=true',
        '-dSubsetFonts=true',
        `-sOutputFile=${outputPath}`,
        inputPath
      ]);

      const compressedBuffer = await fs.readFile(outputPath);

      res.set('Content-Type', 'application/pdf');
      res.set('Content-Disposition', 'attachment; filename="compressed.pdf"');
      return res.status(200).send(compressedBuffer);

    } finally {
      // Clean up temp files — always, even on error
      await fs.unlink(inputPath).catch(() => {});
      await fs.unlink(outputPath).catch(() => {});
    }
  }
);
```

**Confidence:** HIGH for the general structure. MEDIUM for `req.rawBody` vs `req.pipe(bb)` — both paths are included since Firebase Functions v2 behavior with multipart varies between emulator and production (see Pitfalls).

### Pattern 3: Frontend — Replace compressFileHeavy() with Fetch Call

**What:** In `index.html`, replace the client-side JPEG re-render pipeline with a `fetch` call to the deployed Cloud Function.
**When to use:** Frontend integration — the only change to `index.html` for this phase.

```javascript
// Replace the compressFileHeavy() function call path in compressAllFiles():
// Before (client-side JPEG re-render, destroys text):
//   compressedBytes = await compressFileHeavy(arrayBuffer, quality, progressCallback);
//
// After (server-side Ghostscript, preserves text):

const COMPRESS_PDF_ENDPOINT = 'https://us-central1-YOUR_PROJECT_ID.cloudfunctions.net/compressPdf';
// OR during local development: 'http://127.0.0.1:5001/YOUR_PROJECT_ID/us-central1/compressPdf'

async function compressFileServer(file, preset) {
    const formData = new FormData();
    formData.append('pdf', file);         // File object — browser sets Content-Type automatically
    formData.append('preset', preset);    // 'best' | 'balanced' | 'compressed'

    const response = await fetch(COMPRESS_PDF_ENDPOINT, {
        method: 'POST',
        body: formData
        // Do NOT set Content-Type header — browser must set it with boundary automatically
    });

    if (!response.ok) {
        const err = await response.json().catch(() => ({ error: response.statusText }));
        throw new Error(err.error || `Server error: ${response.status}`);
    }

    const arrayBuffer = await response.arrayBuffer();
    return new Uint8Array(arrayBuffer);  // Same type as existing Fast mode returns
}
```

### Anti-Patterns to Avoid

- **Using Multer for file upload parsing:** Multer is broken in Firebase Cloud Functions because the platform pre-parses the request body before Multer runs. Always use Busboy.
- **Setting Content-Type manually in the browser fetch call:** `new FormData()` with `fetch` sets the multipart boundary automatically. If you manually set `Content-Type: multipart/form-data`, the boundary is omitted and Busboy cannot parse the upload.
- **Using Ghostscript `-dPDFSETTINGS=/default`:** This is a valid Ghostscript preset but behaves inconsistently — it does NOT guarantee no compression. Use only `/prepress`, `/ebook`, or `/screen` for predictable output.
- **Not cleaning up `/tmp` files:** Cloud Functions instances reuse across invocations. Orphaned temp files accumulate, eventually filling the 512 MB `/tmp` in-memory filesystem. Always use try/finally to delete both input and output temp files.
- **Relying on `/tmp` persistence across invocations:** `/tmp` is instance-local and in-memory. It may or may not persist between cold start and warm invocations. Always write fresh temp files for each request.
- **Not using `req.rawBody`:** In Firebase Cloud Functions, `req.rawBody` contains the raw request body buffer. Piping `req` to Busboy works on some setups but not all. Prefer `bb.end(req.rawBody)` with a fallback `req.pipe(bb)`.
- **Forgetting `dNOPAUSE`, `dBATCH`, `dQUIET` flags:** Without these, Ghostscript may pause waiting for user input, which will hang the Cloud Function until timeout.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Multipart form-data parsing | Custom request body parser | Busboy | Multer is broken in Cloud Functions; hand-rolling multipart parsing is complex (boundaries, chunking, encoding) |
| PDF image compression | Custom Node.js image resampling | Ghostscript system binary | Ghostscript handles ICC color profiles, CMYK, embedded objects, encryption markers — all edge cases that a custom solution misses |
| Ghostscript binary bundling | Downloading/bundling gs binary | System `/usr/bin/gs` (already in runtime) | Ghostscript 9.55.0 is confirmed in the Ubuntu 22.04 Cloud Functions runtime. Bundling adds 50+ MB to the deployment unnecessarily. |
| CORS handling | Manual Access-Control-Allow-Origin headers | `cors: true` in `onRequest` options | Firebase 2nd gen `onRequest` has a native `cors` option; using it correctly avoids header conflicts |

**Key insight:** Ghostscript is already installed in the Cloud Functions runtime. The entire server-side compression logic is: write temp file → `execFile('gs', args)` → read output → delete temp files. There is no complex library to configure or binary to manage.

## Common Pitfalls

### Pitfall 1: Blaze Plan Required for Cloud Functions

**What goes wrong:** `firebase deploy --only functions` fails with "Cloud Functions requires the Blaze (pay-as-you-go) billing plan."

**Why it happens:** Firebase's free Spark plan does not allow deploying Cloud Functions. The Blaze plan is required even if you never exceed the free tier limits.

**How to avoid:** Before starting implementation, upgrade the Firebase project to Blaze plan in the Firebase console (requires a credit card). The free tier limits (2M invocations/month) are extremely generous for internal tool usage — cost is effectively $0 for low-volume use.

**Warning signs:** Deployment error mentioning billing plan during `firebase deploy`.

### Pitfall 2: Multer Breaks on Cloud Functions

**What goes wrong:** Switching from `busboy` to `multer` for file parsing results in `req.file` and `req.files` being empty/undefined.

**Why it happens:** Firebase Cloud Functions pre-parses the HTTP request body before passing it to your function. Multer tries to read the raw stream that has already been consumed.

**How to avoid:** Always use `busboy`. Pass `req.rawBody` to `busboy.end()`. This is a Cloud Functions-specific constraint, not a general Node.js behavior.

**Warning signs:** `req.file` is `undefined`, `req.body` is `{}` even with form data.

### Pitfall 3: `req.rawBody` Availability in v2 vs Emulator

**What goes wrong:** `req.rawBody` is `undefined` in the local emulator but works in production.

**Why it happens:** The local Functions emulator does not always populate `req.rawBody`. Production Cloud Functions always provide it.

**How to avoid:** Include both paths:
```javascript
if (req.rawBody) {
    bb.end(req.rawBody);
} else {
    req.pipe(bb);
}
```
This handles both emulator (streams) and production (rawBody).

**Warning signs:** "No PDF file received" error only in local emulator, works after deploy.

### Pitfall 4: Ghostscript Hangs Without `-dNOPAUSE -dBATCH`

**What goes wrong:** The Cloud Function runs until its timeout (300 seconds) and then fails, but Ghostscript was not returning an error.

**Why it happens:** Without `-dNOPAUSE` and `-dBATCH`, Ghostscript pauses after each page waiting for a keypress, and displays an interactive prompt at the end of the job. In a non-interactive subprocess, this hangs indefinitely.

**How to avoid:** Always include `-q -dNOPAUSE -dBATCH -dSAFER` in every Ghostscript invocation. These flags make Ghostscript run headless and exit cleanly.

**Warning signs:** Function times out consistently; no error from `execFile`.

### Pitfall 5: Content-Type Header Conflict in fetch() Call

**What goes wrong:** `fetch()` returns 400 or Busboy cannot find files in the parsed request.

**Why it happens:** Developer adds `headers: { 'Content-Type': 'multipart/form-data' }` to the `fetch` options. This overrides the browser's automatic Content-Type which includes the `boundary=` parameter. Without the boundary, Busboy cannot parse the multipart body.

**How to avoid:** Never set Content-Type in the `fetch` options when sending `FormData`. Let the browser set it automatically.
```javascript
// WRONG:
fetch(url, { method: 'POST', body: formData, headers: { 'Content-Type': 'multipart/form-data' } });

// CORRECT:
fetch(url, { method: 'POST', body: formData });
```

**Warning signs:** Busboy receives no files, `pdfBuffer` is null, function returns 400.

### Pitfall 6: Concurrent Requests Collide on Same Temp File Path

**What goes wrong:** Two simultaneous compressions corrupt each other's output files.

**Why it happens:** Using a static filename like `/tmp/input.pdf` means two concurrent requests write to and read from the same file.

**How to avoid:** Always generate unique temp file names per request:
```javascript
const id = crypto.randomUUID();
const inputPath = path.join(os.tmpdir(), `${id}-input.pdf`);
const outputPath = path.join(os.tmpdir(), `${id}-output.pdf`);
```

**Warning signs:** Intermittently corrupted output PDFs under concurrent load.

### Pitfall 7: Request Size Limit for Large PDFs

**What goes wrong:** Large PDFs (over 10–32 MB) fail with HTTP 413 or a Cloud Functions payload error.

**Why it happens:** Cloud Functions impose HTTP request body size limits. Sources conflict: the Busboy article states 10 MB, Firebase AI Logic docs mention 20 MB. The actual limit for Cloud Functions 2nd gen HTTP triggers requires empirical verification.

**How to avoid:**
- The project already has an 80 MB per-file block in `addCompressFiles()`. This may need to be reduced for server mode.
- Recommendation: Set busboy `limits: { fileSize: 50 * 1024 * 1024 }` and add a client-side gate for server mode specifically (e.g., 32 MB max for server, 80 MB for fast mode).
- If limits are genuinely too small, the fallback is to have the client upload to Cloud Storage and have the function read from a signed URL instead of from the request body.

**Warning signs:** HTTP 413 errors, `PayloadTooLargeError` in function logs.

### Pitfall 8: Cold Start Latency

**What goes wrong:** The first invocation after a period of inactivity takes 2–5 seconds before the function starts executing.

**Why it happens:** Cloud Functions spin down when not in use and spin up on demand ("cold start"). The instance initialization time adds latency.

**How to avoid:** This is expected and acceptable for an internal tool. Users see a spinner during compression. For very cold starts, Ghostscript execution itself takes several more seconds, so cold start is not the dominant latency. If it becomes a problem, Cloud Functions 2nd gen supports `minInstances: 1` to keep a warm instance.

**Warning signs:** First request takes noticeably longer; subsequent requests are faster.

function parsePdfUpload(req) {

### Frontend Fetch with FormData (browser side)

```javascript
// Source: MDN Fetch API + FormData API — both stable Web Platform APIs
// Confidence: HIGH

async function compressFileServer(file, preset) {
  const ENDPOINT = window.COMPRESS_PDF_ENDPOINT; // configured per environment

  const formData = new FormData();
  formData.append('pdf', file);       // File object from input[type=file]

      const errBody = await response.json();
      errorMsg = errBody.error || errorMsg;

    throw new Error(errorMsg);

  return new Uint8Array(arrayBuffer); // Consistent with existing compressFast() return type
}
```

## Open Questions

1. **Firebase request body size limit — exact value**
   - What we know: Sources cite 10 MB (Busboy article), 20 MB (Firebase AI Logic docs), and others suggest the limit is enforced by the GCF infrastructure at 32 MB. The app currently allows 80 MB PDFs in Fast mode.
   - What's unclear: The exact HTTP request body limit for Cloud Functions 2nd gen HTTP triggers.
   - Recommendation: During implementation, test with a 15 MB PDF. If 413 errors occur, gate server mode at a lower limit (e.g., 32 MB). Add a client-side check before calling the endpoint: `if (file.size > MAX_SERVER_SIZE) { /* fallback or error */ }`. Keep the 80 MB gate for Fast mode (which stays client-side).

2. **`REQUIREMENTS.md` "files never leave device" — privacy stance for server mode**
   - What we know: REQUIREMENTS.md lists "Server-side processing — Violates core privacy guarantee — files must never leave device" under "Out of Scope."
   - What's unclear: The user explicitly requested server-side Ghostscript compression, overriding this requirement. The planner must decide whether to add a new requirement entry or treat this as an intentional override.
   - Recommendation: Add a COMP-09 requirement to REQUIREMENTS.md: "Server mode (Ghostscript): PDF is sent to Firebase Cloud Function for compression; user is shown a disclosure before upload" — or update the mode copy to state "files are temporarily sent to our server for processing." This is a product/UX decision, not a technical one.

3. **Whether to keep client-side Heavy mode as a fallback**
   - What we know: The client-side `compressFileHeavy()` function will still exist in the codebase after Phase 02.
   - What's unclear: Should Phase 02.1 fully replace it (remove from UI) or add a "server unavailable" fallback path?
   - Recommendation: For simplicity, fully replace — remove the JPEG re-render path from the UI. If the server call fails, show the error in the results list (same as encrypted PDF errors). A fallback to the old approach would confuse users about text selectability.

4. **Endpoint URL configuration — hardcoded vs. environment variable**
   - What we know: `index.html` is a single-file SPA with no build step. Environment variables are not natively supported.
   - What's unclear: How to configure the Cloud Function URL without a build step (development vs. production endpoint).
   - Recommendation: Define a `window.COMPRESS_PDF_ENDPOINT` constant near the top of the `<script>` block in `index.html`, with a comment to update it after deployment. For local development, change the value to the emulator URL (`http://127.0.0.1:5001/...`). This is simple and consistent with the "no build toolchain" project philosophy.

5. **Firebase project setup — user has never used Firebase before**
   - What we know: User is starting fresh with Firebase (per the additional context).
   - What's unclear: Whether the planner should include step-by-step Firebase console instructions (project creation, Blaze upgrade, service account) in the plan.
   - Recommendation: Yes — include Firebase setup as Wave 0 tasks. The plan must cover: create Firebase project in console, upgrade to Blaze plan, install Firebase CLI, `firebase login`, `firebase init functions`. These are prerequisites before any coding.
